# 오늘의 CS 지식

## 주제

연결 리스트

## 카테고리

`algorithm`

## 핵심 요약

- 연결 리스트는 각 노드가 데이터와 다음 노드의 참조를 보관해 순서를 표현하는 선형 자료구조입니다.
- 인덱스로 바로 접근할 수 없으므로 특정 위치 조회는 `O(n)`이지만, 이미 노드를 알고 있다면 연결 변경만으로 삽입·삭제를 `O(1)`에 할 수 있습니다.
- 배열 기반 `ArrayList`는 임의 접근과 순회에 강하고, 연결 리스트는 중간 삽입·삭제가 매우 빈번하며 위치를 이미 알고 있는 경우에 유리합니다.
- Java의 `LinkedList`는 이중 연결 리스트지만, 대부분의 큐·스택 용도에는 더 가볍고 빠른 `ArrayDeque`가 권장됩니다.

## 면접 답변 예시

### 연결 리스트를 한 문장으로 설명해주세요.

연결 리스트는 데이터와 다음 노드의 참조를 가진 노드들을 연결해 순서를 표현하는 자료구조입니다. 배열처럼 인덱스로 즉시 접근하지는 못해 조회는 `O(n)`이지만, 삽입·삭제할 노드의 위치를 이미 알고 있다면 요소 이동 없이 참조만 바꾸므로 `O(1)`에 처리할 수 있습니다.

### 연결 리스트가 필요한 이유는 무엇인가요?

배열은 중간에 원소를 넣거나 빼면 뒤 원소를 이동해야 합니다. 반면 연결 리스트는 노드를 이어 주는 참조만 바꾸므로, 이미 삽입·삭제 지점을 찾은 상태에서 구조 변경 비용을 줄일 수 있습니다. 다만 그 지점을 찾는 탐색 자체가 `O(n)`이므로, 단순히 “삽입이 빠르다”라고 말하면 부족합니다.

### 연결 리스트의 장점과 단점은 무엇인가요?

장점은 앞뒤 노드를 가리키는 참조만 변경해, 위치를 알고 있는 삽입·삭제를 빠르게 수행한다는 점입니다. 크기 변경 시 연속 메모리를 다시 확보할 필요도 없습니다.

반면 임의 접근은 처음 또는 끝에서 순차 탐색해야 하고, 노드마다 참조 필드가 필요합니다. 메모리에 흩어진 노드를 따라가므로 CPU 캐시 효율도 배열보다 대체로 낮습니다.

### 연결 리스트와 비슷한 개념을 비교해서 설명해주세요.

`ArrayList`는 연속 배열을 사용해 `get(i)`가 `O(1)`이고 순회가 빠르지만, 중간 삽입·삭제 시 원소 이동이 필요합니다. `LinkedList`는 이중 연결 구조라 양 끝 작업은 효율적이지만 임의 접근은 느립니다. 큐·스택처럼 양 끝에서만 작업한다면 노드 할당 비용이 적은 `ArrayDeque`가 일반적으로 더 좋은 선택입니다.

### 실무에서 연결 리스트를 사용할 때 주의할 점은 무엇인가요?

Java에서 `LinkedList`를 “삽입이 많으니 빠를 것”이라는 이유만으로 선택하지 않습니다. 실제 워크로드가 임의 조회·전체 순회 중심이면 `ArrayList`가 더 나을 가능성이 큽니다. 큐/덱은 `ArrayDeque`, 스레드 안전한 블로킹 큐는 `LinkedBlockingQueue`·`ArrayBlockingQueue`처럼 요구 사항에 맞는 구현을 선택합니다.

## 실무 관점

- 중간 삽입·삭제의 복잡도는 **삽입 위치를 이미 알고 있을 때**만 `O(1)`입니다.
- 대량 순회와 임의 접근이 많다면 연속 메모리 구조인 `ArrayList`가 보통 유리합니다.
- FIFO/LIFO 용도는 `LinkedList`보다 `ArrayDeque`를 기본 선택지로 검토합니다.

## 코드 예시

```java
Deque<String> queue = new ArrayDeque<>();
queue.offerLast("first");
queue.offerLast("second");

System.out.println(queue.pollFirst()); // first
```

## 상세 설명

### 배열과의 복잡도 비교

| 연산 | `ArrayList` | `LinkedList` |
| --- | --- | --- |
| 인덱스 조회 | `O(1)` | `O(n)` |
| 끝 삽입 | 평균 `O(1)` | `O(1)` |
| 중간 삽입/삭제 | `O(n)` (원소 이동) | 탐색 포함 `O(n)`, 노드를 알고 있으면 `O(1)` |
| 순회 | 캐시 지역성 때문에 대체로 유리 | 참조를 따라 순차 이동 |

단일 연결 리스트는 다음 노드만 가리켜 메모리는 덜 쓰지만 이전 노드로 이동할 수 없습니다. 이중 연결 리스트는 이전·다음 노드를 모두 가리켜 양방향 이동과 삭제는 편하지만 노드당 참조 하나가 더 필요합니다. Java의 `LinkedList`는 이중 연결 리스트입니다.

# 오늘의 Spring 백엔드 지식

## 주제

요청 검증과 예외 응답 설계

## Spring 핵심 요약

- 백엔드 API는 정상 흐름보다 실패 흐름을 일관되게 설계하는 것이 더 중요할 때가 많습니다.
- Spring에서는 Bean Validation과 `@Valid`, `@RestControllerAdvice`를 조합해 검증과 예외 응답을 구성합니다.
- 검증 실패, 권한 실패, 리소스 없음, 비즈니스 규칙 위반을 구분해야 클라이언트와 운영자가 문제를 빠르게 이해할 수 있습니다.

## 동작 원리

1. Controller가 요청 DTO를 바인딩합니다.
2. `@Valid` 또는 `@Validated`가 Bean Validation 제약 조건을 검사합니다.
3. 검증 실패나 비즈니스 예외가 발생하면 예외가 상위로 전파됩니다.
4. `@RestControllerAdvice`가 예외 타입별로 HTTP 상태 코드와 에러 응답을 만듭니다.

## 내부 구성 요소

| 구성 요소 | 역할 |
| --- | --- |
| Bean Validation | DTO 제약 조건을 선언적으로 검증합니다. |
| BindingResult | 검증 오류 정보를 담습니다. |
| RestControllerAdvice | 예외를 공통 API 응답으로 변환합니다. |
| ErrorResponse | 클라이언트가 이해할 수 있는 에러 계약입니다. |

## 실무 구현 포인트

| 상황 | 권장 방식 | 이유 |
| --- | --- | --- |
| 실무 적용 | 요청 DTO와 응답 DTO를 분리합니다 | 일관된 구조와 유지보수성을 높이기 위해서입니다. |
| 실무 적용 | Controller 파라미터에 `@Valid`를 붙여 입력을 검증합니다 | 일관된 구조와 유지보수성을 높이기 위해서입니다. |
| 실무 적용 | `@RestControllerAdvice`에서 예외 타입별 응답을 통일합니다 | 일관된 구조와 유지보수성을 높이기 위해서입니다. |
| 에러 응답에 | 내부 스택트레이스나 민감 정보를 담지 않습니다 | 책임과 변경 범위를 명확히 하기 위해서입니다. |

## 주의사항

- 검증 실패와 비즈니스 예외를 같은 의미로 처리하지 않습니다.
- 에러 응답에 스택트레이스나 민감 정보를 노출하지 않습니다.
- HTTP 상태 코드와 비즈니스 에러 코드를 일관되게 분리합니다.
- 클라이언트가 후속 조치를 할 수 있는 메시지와 코드를 제공합니다.

## 코드 예시

```java
record CreateUserRequest(
        @NotBlank String name,
        @Email String email
) {
}

@PostMapping("/users")
UserResponse create(@Valid @RequestBody CreateUserRequest request) {
    return userService.create(request);
}
```

## Spring 면접 답변 예시

### Q. Spring에서 요청 DTO 검증은 어떻게 하나요?

Spring에서는 요청 DTO에 Bean Validation 애노테이션을 붙이고 Controller에서 `@Valid`로 검증을 수행합니다. 검증 실패나 비즈니스 예외는 `@RestControllerAdvice`에서 공통 에러 응답으로 변환해 API 응답 형식을 일관되게 유지합니다. 상태 코드는 원인을 기준으로 구분해야 하며, 클라이언트 입력 오류는 400, 인증 실패는 401, 권한 부족은 403, 리소스 없음은 404로 표현하는 식으로 설계합니다.

### Q. 실무에서 요청 검증과 예외 응답 설계을 적용할 때 무엇을 주의해야 하나요?

검증 실패와 비즈니스 예외를 같은 의미로 처리하지 않습니다. 에러 응답에 스택트레이스나 민감 정보를 노출하지 않습니다.

## 함께 알아야 하는 개념

| --- | --- |
| Bean Validation | DTO 필드 검증을 선언적으로 표현합니다. |

| HTTP Status | 프로토콜 수준의 실패 의미를 전달합니다. |
| Problem Details | 표준화된 API 에러 응답 형식으로 확장할 수 있습니다. |


## 상세 설명

- DTO 필드에는 `@NotBlank`, `@NotNull`, `@Size`, `@Email` 같은 제약 조건을 둘 수 있습니다.

- 검증 실패는 보통 400 Bad Request로 처리하고, 없는 리소스는 404, 권한 문제는 401 또는 403으로 구분합니다.

- 비즈니스 예외는 도메인 의미가 드러나는 커스텀 예외와 에러 코드를 두면 관리하기 쉽습니다.
